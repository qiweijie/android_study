## 实验一：systrace / Perfetto

### 1.1 Perfetto — 现代化 Trace 工具

```bash
# 方式一：使用 Perfetto UI 在线抓取
# 打开 https://ui.perfetto.dev → Record new trace → 连接设备

# 方式二：命令行抓取
adb shell perfetto -o /data/misc/perfetto-traces/trace.perfetto-trace \
    -t 10s \
    sched freq idle am wm gfx view binder_driver

# 拉取到本地分析
adb pull /data/misc/perfetto-traces/trace.perfetto-trace .

# 用 Perfetto UI 打开分析
# → 上传到 https://ui.perfetto.dev
```

### 1.2 systrace（传统方式）

```bash
# systrace 位于 Android SDK 中
python $ANDROID_HOME/platform-tools/systrace/systrace.py \
    -t 5 \
    -o trace.html \
    sched gfx view wm am app

# 关注的 trace 类别：
# sched  — CPU 调度（线程切换、运行时间）
# gfx    — 图形渲染（帧率、掉帧）
# view   — View 绘制
# wm     — 窗口管理
# am     — Activity 管理
# app    — 应用自定义 trace
# binder_driver — Binder 事务
```

### 1.3 分析要点

```
在 Perfetto/systrace 中关注：

1. CPU 调度
   → 线程是否被频繁抢占
   → 是否有长时间 Sleeping/Uninterruptible

2. 主线程（UI Thread）
   → 是否有超过 16ms 的长帧
   → 卡顿原因：锁等待 / IO / Binder 调用

3. Binder 调用
   → 跨进程调用耗时
   → 是否有 Binder 阻塞

4. 帧率分析
   → SurfaceFlinger 的合成时间
   → 掉帧位置和原因
```

## 实验二：dumpsys 全家桶

### 2.1 内存分析 — dumpsys meminfo

```bash
# 查看系统整体内存
adb shell dumpsys meminfo

# 查看特定进程
adb shell dumpsys meminfo <pid>
adb shell dumpsys meminfo <package_name>

# 关键字段解读：
# Pss Total  — 按比例分配的内存（最常用指标）
# Private Dirty — 进程独占的脏页
# Private Clean — 进程独占的干净页
# Heap Size — Java 堆大小
# Heap Alloc — Java 堆已分配
# Native Heap — Native 层堆内存

# 更详细的内存映射
adb shell dumpsys meminfo -a <package_name>
```

### 2.2 CPU 分析 — dumpsys cpuinfo

```bash
# 查看 CPU 使用情况
adb shell dumpsys cpuinfo

# 输出示例：
# Load: 2.5 / 1.8 / 1.2
# CPU usage from 10s to 0s ago:
#   45% 500/system_server: 30% user + 15% kernel / faults: 2000 minor
#   12% 12345/com.example.app: 8% user + 4% kernel
#   ...
# Total: 85% user + 15% kernel

# 稳定性关注点：
# → system_server CPU 是否过高
# → 某个应用是否异常占用 CPU
# → iowait 是否过高
```

### 2.3 电池分析 — dumpsys batterystats

```bash
# 查看电池统计
adb shell dumpsys batterystats

# 重置统计后重新收集
adb shell dumpsys batterystats --reset

# 关注：
# → 唤醒锁（WakeLock）持有时间
# → 网络使用量
# → CPU 占用时间
```

### 2.4 进程统计 — dumpsys procstats

```bash
# 查看进程运行统计
adb shell dumpsys procstats

# 过去 3 小时
adb shell dumpsys procstats --hours 3

# 关注：
# → 各进程在前台/后台/缓存状态的时间占比
# → PSS 内存使用趋势
```

### 2.5 综合练习

```bash
# 练习 1：找出当前内存占用最多的 3 个进程
adb shell dumpsys meminfo | head -30

# 练习 2：查看 system_server 的详细内存分布
adb shell dumpsys meminfo system_server

# 练习 3：查看 CPU 使用排行
adb shell dumpsys cpuinfo | head -20

# 练习 4：查看某应用的完整资源使用
adb shell dumpsys meminfo <your_app_package>
```

## 实验三：bugreport

### 3.1 生成 bugreport

```bash
# 生成 bugreport（需要几分钟）
adb bugreport ./bugreport.zip

# bugreport 包含：
# ├── bugreport-xxx.txt     ← 主文件（系统状态快照）
# ├── version.txt
# ├── dumpstate_board.txt
# ├── FS/                   ← 文件系统快照
# │   ├── data/anr/         ← ANR traces
# │   ├── data/tombstones/  ← Tombstone 文件
# │   └── proc/             ← /proc 信息
# └── proto/                ← Protocol Buffer 数据
```

### 3.2 bugreport 分析要点

```
bugreport-xxx.txt 中的关键段落：

1. SYSTEM LOG (logcat)
   → 搜索 "FATAL", "ANR", "WATCHDOG"

2. EVENT LOG
   → am_crash / am_anr / am_proc_died 事件

3. DUMPSYS (各服务状态)
   → dumpsys activity / window / meminfo

4. ANR TRACES (/data/anr/)
   → 线程堆栈快照

5. TOMBSTONES (/data/tombstones/)
   → Native Crash 信息

6. KERNEL LOG (dmesg)
   → OOM / panic / driver 错误
```

### 3.3 用 Battery Historian 分析

```bash
# Battery Historian 是 Google 开源的 bugreport 可视化工具
# 安装：
docker run -p 9999:9999 gcr.io/android-battery-historian/stable:latest

# 打开浏览器 http://localhost:9999
# 上传 bugreport.zip → 可视化分析
```

## 实验四：addr2line / ndk-stack

### 4.1 addr2line — 地址转源码行号

```bash
# 当你有一个 Native Crash 的地址：
# backtrace:
#   #00 pc 0001a234 /system/lib64/libxxx.so

# 使用 addr2line 还原：
# 需要带符号的 .so 文件（编译时生成的未 strip 版本）
addr2line -C -f -e libxxx.so 0x0001a234

# 参数说明：
# -C  解码 C++ 符号名（demangle）
# -f  显示函数名
# -e  指定带符号的 .so 文件

# Android NDK 自带的 addr2line：
$NDK/toolchains/llvm/prebuilt/linux-x86_64/bin/llvm-addr2line \
    -C -f -e ./symbols/system/lib64/libxxx.so 0x0001a234

# 批量符号化多个地址：
echo "0x1234 0x5678 0x9abc" | \
    xargs -n1 addr2line -C -f -e libxxx.so
```

### 4.2 ndk-stack — 自动化符号化

```bash
# ndk-stack 可以自动处理整个 Tombstone 文件
# 需要指定符号目录（包含未 strip 的 .so 文件）

# 从 Tombstone 文件：
adb shell cat /data/tombstones/tombstone_00 | \
    ndk-stack -sym ./symbols/

# 从 logcat：
adb logcat -b crash | ndk-stack -sym ./symbols/

# symbols 目录结构应该是：
# symbols/
# ├── system/
# │   ├── lib64/
# │   │   ├── libxxx.so     (带符号)
# │   │   └── libbinder.so  (带符号)
# │   └── bin/
# └── vendor/
```

### 4.3 llvm-symbolizer

```bash
# LLVM 版本的符号化工具（更现代）
llvm-symbolizer --obj=libxxx.so 0x1a234

# 支持更多输出格式
llvm-symbolizer -print-source-context-lines=5 --obj=libxxx.so 0x1a234
```

### 4.4 实操练习

```bash
# 步骤 1：用 Day4 的 JNI Demo 触发一个 Native Crash
# 步骤 2：收集 Tombstone
adb shell ls /data/tombstones/
adb pull /data/tombstones/tombstone_00 .

# 步骤 3：查看 Crash 日志
adb logcat -b crash -d | tail -30

# 步骤 4：符号化
# 找到编译产物中带符号的 .so
# 通常在 app/build/intermediates/cmake/ 下
ndk-stack -sym ./app/build/intermediates/cmake/debug/obj/arm64-v8a/ \
    -dump tombstone_00
```
