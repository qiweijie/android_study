### 1.2 分析步骤

```
打开 https://ui.perfetto.dev/ 加载 trace 文件

分析流程：
1. 找到目标进程
   → Ctrl+F 搜索进程名
   → 展开该进程的所有线程

2. 定位卡顿帧
   → 找 "Frame Timeline" 泳道
   → 红色帧 = 超时帧
   → 点击红色帧查看耗时

3. 分析主线程
   → 展开 main 线程
   → 查看该帧时间段内的调用栈
   → 常见阻塞原因：
      ├── Binder 调用耗时长 → 优化跨进程调用
      ├── Lock contention → 锁竞争，减少同步范围
      ├── IO 操作 → 移到子线程
      └── 大量 measure/layout → 优化布局层级

4. 检查调度延迟
   → Scheduling 泳道
   → "Runnable" 状态持续时间长 = CPU 抢不到
   → 检查是否有高优先级线程抢占
```

## 实验二：simpleperf CPU Profiling 实战

### 2.1 录制与分析

```bash
# 获取目标进程 PID
PID=$(adb shell pidof com.example.app)
echo "Target PID: $PID"

# 录制 CPU profile（带调用栈，10 秒）
adb shell simpleperf record -p $PID \
    -g --duration 10 \
    --call-graph dwarf \
    -f 1000 \
    -o /data/local/tmp/perf.data

# 拉取数据
adb pull /data/local/tmp/perf.data .

# 文本报告
simpleperf report -i perf.data --sort comm,dso,symbol

# 生成可视化 HTML 报告
python simpleperf/report_html.py -i perf.data -o report.html
```

### 2.2 火焰图生成

```bash
# 方法一：simpleperf 内置
simpleperf report-sample -i perf.data --show-callchain > out.perf

# 方法二：使用 FlameGraph 工具
git clone https://github.com/brendangregg/FlameGraph.git
simpleperf report-sample -i perf.data | \
    FlameGraph/stackcollapse-perf.pl | \
    FlameGraph/flamegraph.pl > flamegraph.svg

# 在浏览器中打开 flamegraph.svg
```

### 2.3 分析要点

```
火焰图阅读方法：
  y 轴 = 调用栈深度（底部是入口，顶部是叶子函数）
  x 轴 = 采样占比（越宽 = CPU 占用越多）

重点关注：
1. 最宽的"平顶"函数 → 自身 CPU 消耗最多
2. 调用链中的宽函数 → 热点路径
3. GC 相关函数占比 → 是否有过多内存分配
4. Binder 相关函数占比 → 是否有频繁 IPC
```

## 实验三：内存分析全链路

### 3.1 内存趋势监控

```bash
# 创建内存趋势监控脚本 mem_monitor.sh
# 每 5 秒采集一次，共 60 次（5 分钟）

PID=$(adb shell pidof com.example.app)
echo "Monitoring PID: $PID"
echo "Time,NativeHeap,DalvikHeap,PssTotal" > mem_trend.csv

for i in $(seq 1 60); do
    TIMESTAMP=$(date +%H:%M:%S)
    
    # 解析 dumpsys meminfo
    MEMINFO=$(adb shell dumpsys meminfo $PID)
    
    NATIVE=$(echo "$MEMINFO" | grep "Native Heap" | head -1 | awk '{print $3}')
    DALVIK=$(echo "$MEMINFO" | grep "Dalvik Heap" | head -1 | awk '{print $3}')
    TOTAL=$(echo "$MEMINFO" | grep "TOTAL" | head -1 | awk '{print $2}')
    
    echo "$TIMESTAMP,$NATIVE,$DALVIK,$TOTAL" >> mem_trend.csv
    echo "[$i/60] $TIMESTAMP - Native:${NATIVE}K Dalvik:${DALVIK}K Total:${TOTAL}K"
    
    sleep 5
done

echo "Done! Data saved to mem_trend.csv"
```

### 3.2 showmap 详细分析

```bash
# 查看进程的完整内存映射
adb shell showmap $(adb shell pidof com.example.app)

# 关注的字段：
# virtual   shared   shared  private  private
#    size    clean    dirty    clean    dirty    object
# -------- -------- -------- -------- --------
#   ...
#  重点关注 private dirty 最大的几个映射：
#  [anon:libc_malloc]  → Native 堆
#  /dev/ashmem/dalvik  → Dalvik 堆
#  *.so                → so 库的内存

# 分析 smaps 获取更详细的信息
adb shell cat /proc/$(adb shell pidof com.example.app)/smaps | \
    grep -E "^[0-9a-f]|Rss:|Pss:|Private_Dirty:" | head -100
```

### 3.3 Native 内存泄漏检测

```bash
# 步骤 1：启用 malloc_debug
adb root
adb shell setprop libc.debug.malloc.options "backtrace leak_track"
adb shell setprop libc.debug.malloc.program com.example.app

# 步骤 2：重启应用
adb shell am force-stop com.example.app
adb shell am start com.example.app/.MainActivity

# 步骤 3：运行一段时间，制造泄漏场景

# 步骤 4：检查泄漏
adb shell dumpsys malloc_info -a $(adb shell pidof com.example.app)

# 输出分析：
# Leaked allocations:
#   size=256 count=1000 total=256000       ← 256 字节分配了 1000 次未释放
#   backtrace:
#     #00 pc 0x1234 libmylib.so (alloc_buffer+16)   ← 分配位置
#     #01 pc 0x2345 libmylib.so (process_data+48)

# 步骤 5：使用 addr2line 定位源码
# (需要带符号的 so 文件)
addr2line -f -e libmylib.so 0x1234

# 步骤 6：清理，关闭 malloc_debug
adb shell setprop libc.debug.malloc.options ""
adb shell setprop libc.debug.malloc.program ""
```

## 实验五：自动化监控原型搭建

### 5.1 DropBox 事件监听脚本

```bash
#!/bin/bash
# dropbox_monitor.sh — DropBox 事件自动监控
# 持续监听 DropBox 中的异常事件并生成报告

OUTPUT_DIR="./stability_report"
mkdir -p $OUTPUT_DIR

echo "=== 稳定性自动监控开始 ==="
echo "输出目录: $OUTPUT_DIR"

# 获取所有 DropBox 条目
echo "--- 采集 DropBox 事件 ---"
ENTRIES=$(adb shell dumpsys dropbox --print 2>/dev/null)

# 分类统计
CRASH_COUNT=$(echo "$ENTRIES" | grep -c "data_app_crash" 2>/dev/null || echo 0)
ANR_COUNT=$(echo "$ENTRIES" | grep -c "data_app_anr" 2>/dev/null || echo 0)
NATIVE_CRASH=$(echo "$ENTRIES" | grep -c "data_app_native_crash" 2>/dev/null || echo 0)
WATCHDOG=$(echo "$ENTRIES" | grep -c "system_server_watchdog" 2>/dev/null || echo 0)
TOMBSTONE=$(echo "$ENTRIES" | grep -c "SYSTEM_TOMBSTONE" 2>/dev/null || echo 0)

# 生成报告
REPORT="$OUTPUT_DIR/report_$(date +%Y%m%d_%H%M%S).txt"
cat > $REPORT << REPORT_END
========================================
   稳定性监控报告
   时间: $(date '+%Y-%m-%d %H:%M:%S')
   设备: $(adb shell getprop ro.product.model)
   版本: $(adb shell getprop ro.build.display.id)
========================================

一、异常事件统计
  Java Crash:      $CRASH_COUNT 次
  Native Crash:    $NATIVE_CRASH 次
  ANR:             $ANR_COUNT 次
  Watchdog:        $WATCHDOG 次
  Tombstone:       $TOMBSTONE 次

二、严重程度评估
REPORT_END

# 评估严重程度
if [ "$WATCHDOG" -gt 0 ]; then
    echo "  [P0 严重] Watchdog 重启 $WATCHDOG 次 — 需要立即排查!" >> $REPORT
fi
if [ "$NATIVE_CRASH" -gt 0 ]; then
    echo "  [P1 高]   Native Crash $NATIVE_CRASH 次 — 需要分析 Tombstone" >> $REPORT
fi
if [ "$CRASH_COUNT" -gt 5 ]; then
    echo "  [P1 高]   Java Crash 频繁 ($CRASH_COUNT 次)" >> $REPORT
fi
if [ "$ANR_COUNT" -gt 3 ]; then
    echo "  [P2 中]   ANR 较多 ($ANR_COUNT 次)" >> $REPORT
fi

echo "" >> $REPORT
echo "三、详细事件列表" >> $REPORT
echo "$ENTRIES" | grep -E "data_app_crash|data_app_anr|data_app_native_crash|system_server" >> $REPORT

echo "报告已生成: $REPORT"
cat $REPORT
```
