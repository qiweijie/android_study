## 一、Perfetto / systrace — 卡顿定位

### 1.1 卡顿的本质

```
Android 渲染管线（16.6ms 一帧 = 60fps）：

主线程 (UI Thread):
  ├── 处理输入事件
  ├── 执行动画
  ├── measure / layout / draw
  └── 同步到 RenderThread

RenderThread:
  ├── 执行 GPU 指令
  └── 提交 Buffer 到 SurfaceFlinger

SurfaceFlinger:
  ├── 合成所有 Layer
  └── 提交到 Display

任何一个环节超过 16.6ms → 掉帧 → 用户感知卡顿
```

### 1.2 Perfetto 分析要点

```
在 Perfetto UI 中关注：

1. Scheduling 泳道
   → 线程在 CPU 上的运行时间
   → Running (绿色) / Sleeping (灰色) / Runnable (蓝色) / Uninterruptible (红色)
   → 红色 = IO 等待，需要关注

2. Frame 时间线
   → 每一帧的开始和结束
   → 超过 16.6ms 的帧标红
   → 点击查看帧内各阶段耗时

3. Binder 调用
   → 跨进程调用的耗时
   → 是否有 Binder 调用阻塞了主线程

4. 锁竞争
   → monitor contention (锁等待)
   → 显示等待了多长时间、等谁的锁
```

## 二、simpleperf — CPU Profiling

### 2.1 基本使用

```bash
# 录制 CPU profile（10 秒）
adb shell simpleperf record -p $(adb shell pidof com.example.app) \
    -g --duration 10 -o /data/local/tmp/perf.data

# 拉取到本地
adb pull /data/local/tmp/perf.data .

# 查看报告
simpleperf report -i perf.data

# 输出示例：
# Overhead  Command     Shared Object      Symbol
# 25.30%   main        libmylib.so        MyClass::heavyWork()
# 12.50%   main        libart.so          art::gc::Heap::Alloc
#  8.20%   Binder:1    libbinder.so       IPCThreadState::transact
```

### 2.2 生成火焰图

```bash
# 使用 FlameGraph 工具
simpleperf report-sample --symfs ./symbols/ -i perf.data | \
    FlameGraph/stackcollapse-perf.pl | \
    FlameGraph/flamegraph.pl > flamegraph.svg

# 或使用 simpleperf 自带的脚本
python simpleperf/report_html.py -i perf.data -o report.html
```

## 三、内存分析

### 3.1 dumpsys meminfo 详解

```bash
adb shell dumpsys meminfo <pid>

# 关键指标：
#                   Pss      Private   Private   SwapPss   Heap    Heap    Heap
#                  Total      Dirty    Clean      Dirty    Size   Alloc    Free
#                ------   --------  --------  --------  ------  ------  ------
#   Native Heap    8000      7900       0         0     16384   12000    4384
#   Dalvik Heap    5000      4800       0         0     10240    8000    2240
#   ...
#   TOTAL         45000     38000     4000       500

# 关键指标解读：
# Pss Total    → 按比例分配的总内存（最常用）
# Private Dirty → 进程独占的已修改内存（最"真实"的占用）
# Native Heap  → C/C++ 层堆内存
# Dalvik Heap  → Java 层堆内存
# Heap Size    → 堆大小上限
# Heap Alloc   → 已分配
# Heap Free    → 空闲
```

### 3.2 showmap / procrank

```bash
# showmap — 详细的内存映射
adb shell showmap <pid>

# procrank — 进程内存排行（需要 root）
adb shell procrank
# 输出：PID → Vss → Rss → Pss → Uss → cmdline
```

### 3.3 内存泄漏的判断标准

```
如何判断是否有内存泄漏：

1. 趋势观察法
   → 多次执行 dumpsys meminfo
   → 如果 Pss/Private Dirty 持续增长且不回落 → 可能泄漏

2. GC 后观察法
   → 手动触发 GC: adb shell kill -10 <pid>
   → 再查看 meminfo，如果内存不减少 → 泄漏

3. hprof 分析法
   → 用 Android Studio Profiler 抓 heap dump
   → 查看 Activity/Fragment 实例数是否持续增长
```

## 四、Native 内存泄漏检测

### 4.1 malloc_debug

```bash
# 启用 malloc_debug
adb shell setprop libc.debug.malloc.options "backtrace leak_track"
adb shell setprop libc.debug.malloc.program com.example.app

# 重启应用，运行一段时间后：
adb shell dumpsys malloc_info -a <pid>

# 输出泄漏的分配信息：
# Leaked allocations:
#   size=1024 count=500 total=512000
#   backtrace:
#     #00 pc 0x12345 libmylib.so (MyClass::alloc+16)
#     #01 pc 0x12abc libmylib.so (MyClass::run+48)
```

### 4.2 libmemunreachable

```bash
# libmemunreachable 检测不可达的 Native 内存
# 在代码中调用：
# #include <memunreachable/memunreachable.h>
# bool success = GetUnreachableMemory(info);

# 或通过 dumpsys:
adb shell dumpsys -t 60 meminfo --unreachable <pid>
```

### 4.3 C++ 开发者工具对比

```
Linux 上          Android 上              说明
──────────────────────────────────────────────────
valgrind          ASan/HWASan            编译时插桩
mtrace            malloc_debug           动态跟踪
LeakSanitizer     libmemunreachable      检测泄漏
massif            dumpsys meminfo        内存分析
perf              simpleperf             CPU profile
strace            strace (root)          系统调用跟踪
```
