### 1.6 流程规范

```
一、Crash 分级响应流程

  发现 Crash
      │
      ▼
  自动分级（P0-P3）
      │
      ├── P0 → 5 分钟内通知 → 2 小时内定位 → 24 小时内修复
      ├── P1 → 1 小时内通知 → 1 天内定位 → 3 天内修复
      ├── P2 → 日报通知 → 1 周内定位 → 2 周内修复
      └── P3 → 周报跟踪 → 排入版本计划
      
  每个 Crash 都需要：
    □ 根因分析报告
    □ 修复方案审核
    □ 回归验证


二、代码审查 Checklist（稳定性相关）

  □ 资源管理
    ├── Native 内存分配是否有对应的释放？
    ├── Java 层是否有未关闭的 Cursor / Stream？
    └── Binder 调用是否有超时保护？
  
  □ 线程安全
    ├── 共享数据是否有适当的同步？
    ├── 锁的获取顺序是否一致？（防死锁）
    └── 是否有在主线程做耗时操作？
  
  □ 异常处理
    ├── 是否有 null 检查？
    ├── 是否有数组越界检查？
    └── 跨进程调用是否处理了 RemoteException？
  
  □ 性能影响
    ├── 是否引入了新的 ANR 风险？
    ├── 内存分配是否合理？
    └── 是否有频繁的 GC 触发？


三、上线前稳定性准入门槛

  版本发布前必须满足：
  
  ┌──────────────────────────────────────────────────┐
  │ 准入项                    │ 标准               │ 
  ├──────────────────────────────────────────────────┤
  │ Java Crash 率             │ ≤ ___‰             │
  │ Native Crash 率           │ ≤ ___‰             │
  │ ANR 率                    │ ≤ ___‰             │
  │ Watchdog 重启             │ 0 次               │
  │ 启动时间 P90              │ ≤ ___ms            │
  │ 72h 内存增长              │ ≤ ___MB            │
  │ Monkey 测试通过率          │ ≥ ___% (10万次事件) │
  │ CTS 兼容性测试            │ 100% 通过          │
  └──────────────────────────────────────────────────┘
  
  不满足准入条件：
    □ 阻塞发布
    □ 升级到 P0 处理
    □ 需要总监审批豁免（记录风险）
```
