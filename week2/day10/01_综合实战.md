## 一、端到端 Crash 分析演练

### 1.1 从 bugreport 开始

```bash
# 抓取 bugreport
adb bugreport bugreport.zip

# 解压后的关键文件：
bugreport-<device>-<datetime>.txt    # 主报告
FS/data/tombstones/                  # Tombstone 文件
FS/data/anr/                         # ANR traces
FS/data/system/dropbox/              # DropBox 事件
version.txt                          # 版本信息
dumpstate_board.txt                  # 硬件状态
```

### 1.2 Bugreport 分析框架

```
bugreport 完整分析流程（六步法）：

Step 1: 确认基本信息
  → 搜索 "Build fingerprint" → 确认版本
  → 搜索 "Kernel version" → 确认内核版本
  → 搜索 "== uptime ==" → 确认运行时间
  → 搜索 "ro.build.type" → userdebug / eng / user

Step 2: 定位异常事件
  → 搜索 "SYSTEM RESTART" → 系统是否重启过
  → 搜索 "FATAL EXCEPTION" → Java Crash
  → 搜索 "Native crash" / "signal" → Native Crash
  → 搜索 "ANR in" → ANR
  → 搜索 "Watchdog" → Watchdog 事件
  → 搜索 "lowmemorykiller" → LMK 杀进程

Step 3: 分析时间线
  → 搜索 "== system log ==" → 进入 system log
  → 根据异常事件时间戳，前后各看 30 秒
  → 构建事件时间线：
     T-30s: 发生了什么？
     T-10s: 有异常日志吗？
     T-0s:  Crash/ANR 发生
     T+5s:  系统如何响应？

Step 4: 深入分析根因
  → Java Crash: 查看完整堆栈，定位到源码行
  → Native Crash: 查看 Tombstone，addr2line 解析
  → ANR: 查看 traces.txt，分析主线程阻塞原因
  → Watchdog: 查看 HandlerChecker 状态

Step 5: 关联分析
  → CPU 使用率: 搜索 "CPU usage"
  → 内存状态: 搜索 "Total RAM" / "Free RAM"
  → Binder 状态: 搜索 "binder_sample" / "binder transactions"
  → IO 状态: 搜索 "iowait"

Step 6: 输出结论
  → 根因（Root Cause）
  → 影响范围
  → 修复建议
  → 预防措施
```

### 1.3 实战案例：复合型问题分析

```
场景：system_server 崩溃后设备自动重启

分析思路：
1. bugreport 中搜索 "system_server" + "died"
   → 确认 system_server 是被 Watchdog kill 还是自身 Crash

2. 如果是 Watchdog kill：
   → 搜索 "Blocked in handler"
   → 找到被阻塞的 HandlerChecker
   → 查看 main 线程堆栈（通常在 traces.txt）

   常见阻塞场景：
   (a) 等 Binder 调用返回
       → "- waiting on <binder>"
       → 需要找到 Binder 对端进程的状态
   
   (b) 等锁
       → "- waiting to lock <0x...> held by thread ..."
       → 跟踪持锁线程在做什么
   
   (c) IO 阻塞
       → "- at java.io.FileOutputStream.write"
       → 检查磁盘是否满了 / IO 是否异常慢

3. 如果是自身 Crash：
   → 搜索 "FATAL EXCEPTION: main" in system_server
   → 查看完整 Java 堆栈
   → 或搜索 "signal 6 (SIGABRT)" → Native Crash

4. 检查事发时的系统状态
   → CPU: 是否 100% → 搜索 "Load:"
   → 内存: 是否 OOM → 搜索 "Low on memory"
   → 磁盘: 是否满了 → 搜索 "No space left"
```

## 二、跨进程死锁分析

### 2.1 死锁的本质

```
Android 系统中跨进程死锁的典型模式：

模式一：Binder 互相等待

Process A (AMS)              Process B (WMS)
─────────────────           ─────────────────
持有 AMS 锁                 持有 WMS 锁
    │                           │
    ├── 发起 Binder 调用 ──────→ 需要获取 WMS 锁
    │   等待 B 返回...          (已持有，可进入)
    │                           │
    │                           ├── 发起 Binder 调用
    │   ← ─────────────────────── 需要获取 AMS 锁
    │   (A 持有 AMS 锁)         等待 A 返回...
    │                           │
    ▼                           ▼
  BLOCKED                    BLOCKED
  → 死锁！                   → 死锁！


模式二：跨进程锁嵌套

Thread-1 (system_server):
  synchronized(mAMSLock) {
      // 调用 WMS 方法（Binder 到 WMS 进程 or 同进程锁）
      mWMS.updateWindow();  // 需要 mWMSLock
  }

Thread-2 (system_server):
  synchronized(mWMSLock) {
      // 调用 AMS 方法
      mAMS.getActivity();  // 需要 mAMSLock
  }

→ 同进程内两个线程交叉持锁 → 死锁


模式三：Binder 线程池耗尽

system_server 的 Binder 线程池（默认 16 个）全部被占满：
  Thread-1: 处理请求 A → 调用 Service B → 等待返回
  Thread-2: 处理请求 A → 调用 Service B → 等待返回
  ...
  Thread-16: 处理请求 A → 调用 Service B → 等待返回
  → 所有 Binder 线程都在等 → 新请求无法处理 → 类似死锁
```

### 2.2 死锁分析方法

```bash
# 1. 查看 Binder 状态
adb shell cat /sys/kernel/debug/binder/transactions
adb shell cat /sys/kernel/debug/binder/failed_transactions
adb shell cat /sys/kernel/debug/binder/transaction_log

# 2. 查看线程堆栈（最重要）
adb shell kill -3 $(adb shell pidof system_server)
# 等待几秒
adb pull /data/anr/traces.txt

# 3. 分析 traces.txt 中的死锁

# 在 traces.txt 中查找死锁信号：
# (a) "waiting to lock" + "held by"
#     → 锁等待链
# (b) "Binder Thread #" + "waiting on"
#     → Binder 调用等待
# (c) 多个线程形成等待环路
#     → 经典死锁
```

### 2.3 traces.txt 死锁分析示例

```
典型的死锁 traces.txt 片段：

"main" prio=5 tid=1 Blocked
  | group="main" sCount=1 dsCount=0 flags=1 obj=0x... self=0x...
  | sysTid=1234 nice=-2 cgrp=default
  | held mutexes=
  at com.android.server.am.ActivityManagerService.startActivity(AMS.java:5678)
  - waiting to lock <0xAAAA> (a WindowManagerGlobalLock) held by thread 25
  at com.android.server.am.ActivityManagerService.handleMessage(AMS.java:2345)

"Binder:1234_3" prio=5 tid=25 Blocked
  | group="main" sCount=1 dsCount=0 flags=1 obj=0x... self=0x...
  | sysTid=1259 nice=0 cgrp=default
  | held mutexes=
  at com.android.server.wm.WindowManagerService.relayoutWindow(WMS.java:3456)
  - waiting to lock <0xBBBB> (a ActivityManagerService) held by thread 1
  at com.android.server.wm.Session.relayout(Session.java:234)

分析：
  Thread main (tid=1):
    持有: (隐含) AMS 锁
    等待: <0xAAAA> WMS 锁 → 被 thread 25 持有
  
  Thread Binder:1234_3 (tid=25):
    持有: <0xAAAA> WMS 锁
    等待: <0xBBBB> AMS 锁 → 被 thread 1 持有
  
  → main → 等 WMS 锁 → 被 tid=25 持有
  → tid=25 → 等 AMS 锁 → 被 main 持有
  → 形成环路 = 死锁！
```

### 2.4 Binder 线程池耗尽排查

```bash
# 查看 Binder 线程状态
adb shell cat /sys/kernel/debug/binder/stats | grep "proc "

# 查看具体进程的 Binder 线程
adb shell cat /sys/kernel/debug/binder/proc/$(adb shell pidof system_server)

# 关注：
# threads: 16    ← 线程池大小
# requested_threads: 0
# requested_threads_started: 15
# ready_threads: 0    ← 0 表示所有线程都忙

# 如果 ready_threads = 0 且 requested_threads_started = max
# → Binder 线程池耗尽
# → 需要查看每个 Binder 线程在做什么（看 traces.txt）
```

### 3.3 从 C++ 到 Android 系统开发的技能对照

```
C++ 通用技能            Android 系统中的对应
──────────────────────────────────────────────────
智能指针 (shared_ptr)  → sp<> / wp<>（libutils）
STL 容器              → Vector / KeyedVector（libutils）
多线程 (std::thread)  → Thread / Looper / Handler（Native）
IPC (socket/pipe)     → Binder（首选）/ socket（低级通信）
构建系统 (cmake)      → Android.bp / Soong
单元测试 (gtest)      → gtest（Android 也用 gtest）
日志 (printf/spdlog)  → ALOG* 宏（liblog）
内存检测 (valgrind)   → ASan / malloc_debug
性能分析 (perf)       → simpleperf
调试 (gdb)            → lldb / gdbclient.py
```
